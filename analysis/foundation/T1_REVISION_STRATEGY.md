# T1 Revision Strategy (Aggregation)

## Course-Level Fixes
1. Establish a cross-module terminology standard:
   - first-use plain-language definitions for critical terms
   - term consistency rules (single preferred term per concept)
2. Establish cross-module example coverage standard:
   - each module should include at least one example mapped to non-technical/general audience and one to technical/analytical audience
   - add compact role-variant callouts instead of full rewrites
3. Add a reusable regulated-context safety overlay:
   - brief note on authorized tools, sensitive data boundaries, and documentation/accountability expectations

## Tier-Level Fixes
1. Create a shared exercise calibration standard for T1:
   - one worked example pattern per module
   - common rubric-anchor style for 1/3/5 scoring where checklists exist
   - explicit "what good looks like" cues
2. Introduce a beginner-safe entry protocol for early modules (T1M01-T1M02):
   - fallback prompt source paths
   - pre-exercise readiness checks (tool access, starting materials)
3. Add lightweight team deployment notes where artefacts naturally scale:
   - owner, approval, update cadence, and team-use boundaries
4. Preserve and replicate T1M03-T1M04 readability patterns across earlier modules:
   - shorter paragraphs, clearer headers, reduced idiomatic phrasing

## Module-Specific Fixes
1. T1M01:
   - reduce low-yield historical pacing drag
   - strengthen beginner onboarding and glossary entry points
2. T1M02:
   - rebalance early technical load and tokenization depth
   - improve practical bridge before heavy mechanics
3. T1M03:
   - add medium/ambiguous risk calibration defaults
   - clarify context-dependent risk handling
4. T1M04:
   - add explicit constraint-writing method based on task failure-mode identification
   - improve prompt-library lifecycle guidance (organization/maintenance)

## Suggested Shared Assets
1. `T1 Shared Glossary` (plain-language, first-use aligned to module references).
2. `T1 Exercise Rubric Pack` (worked examples + scoring anchors).
3. `Example Variant Bank` (short role-variant snippets by domain/persona for reuse across modules).
4. `Prompt/Artefact Governance Mini-Guide` (owner, approval, version, last-tested).
5. `Regulated-Use Safety Note` (authorized tool/data handling checklist).

## Rejected / Deferred Suggestions
1. Rewriting foundation modules into full persona-specific tracks: deferred (high complexity, low maintainability risk).
2. Converting core modules into enterprise procurement/deployment manuals: rejected for core tier scope.
3. Expanding core modules into advanced deep technical paths: deferred to optional/advanced assets.
4. Reordering foundation sequence based on isolated preference signals: deferred pending broader evidence.

## What Must Not Change
1. AI-as-tool, human-accountability philosophy.
2. Framework-centered pedagogy (clear conceptual models per module).
3. Real-work before/after exercise design.
4. Practical artefact outputs as transfer anchors.
5. Progressive flow from understanding -> governance -> structured control.
6. Accessibility gains seen in later modules (clear structure, direct language, practical tone).
