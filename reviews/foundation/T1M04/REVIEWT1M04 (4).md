# 2. Concept Explanation

**Did the explanation help you understand the concept?**

Yes, though I needed to read the section on Role, Context, Task, Constraints, and Output Format twice before I felt fully comfortable with all five elements.  The second read was much faster — I think on the first pass I was absorbing what each element was, and on the second pass I was seeing how they work together.

**Did any terminology feel confusing?**

"Output schema" appeared once in the common weak-prompt patterns list.  The module uses "Output Format" everywhere else and I knew what that meant.  "Schema" felt slightly technical and is the one word I had to infer from context.

"Downstream rework" — I understood this from context (it means having to redo the AI's output before you can use it), but it is professional jargon that might confuse some readers.

**Did the mental model make sense?**

Yes.  The explanation of what each element prevents was particularly helpful:
- Role prevents tone and perspective drift
- Context prevents irrelevant assumptions
- Task prevents ambiguity
- Constraints prevent hallucination and overreach
- Output Format prevents downstream rework

I copied this list into my notes because it gave me a reason to use each element, not just a definition.  Knowing WHY I should include each part makes it much easier to remember to use it.

**Was the explanation too long, too short, or appropriate?**

Appropriate.  Possibly the first module in this course where I did not wish for a shorter or longer explanation — this felt well-calibrated.

Rating (1–5):
Concept clarity: 4
Terminology accessibility: 4
Usefulness of mental model: 5

Comments:

The list of common weak-prompt patterns felt personally accurate in a slightly uncomfortable way:
- "Multi-goal prompts with no priority" — I have definitely asked Copilot to do three things in one message.
- "Missing audience definition" — I did not know this was important.
- "No output schema" — I didn't know what output format to request because I didn't know I could specify it.

Seeing my own habits described as "common weak-prompt patterns" was actually encouraging rather than discouraging, because it told me that my problems are normal and solvable, not signs that I am unusually bad at this.

# 9. Most Valuable Part

The list pairing each prompt element with the failure mode it prevents.

Knowing that Output Format prevents downstream rework changed how I think about every AI interaction.  Before this module, I accepted whatever structure the AI chose to give me.  Now I know that if I need a structured document, I need to specify the document structure in the prompt.  That is a simple insight with immediate practical value.

---

# 10. Weakest Part

The guidance on writing effective constraints.  The module identifies "vague constraints such as 'make it better'" as a mistake and gives three good examples in the finance demonstration and three in the ops demonstration.  But there is no conceptual guidance on what makes a constraint specific and effective.

For a beginner, learning to write constraints that are clear and specific enough to actually control AI behavior requires more guidance than the demonstrations alone provide.  A dedicated section on constraint quality — with two or three pairs of weak vs. strong constraints — would help significantly.

---

# 11. Missing Content

The module is well-structured and did not assume knowledge I lacked.

What would add value for my professional context:

1. **Administrative and support role demonstrations.**  The SOP conversion demo is close, but an explicit administrative prompt example (drafting professional correspondence, preparing briefing documents, summarizing board materials) would make the module even more directly applicable.

2. **Constraint-writing guidance.**  As described above.

3. **A note about AI tool access for the exercise.**  The exercise requires testing in a real AI tool.  Learners should be advised to ensure access before beginning.

4. **Examples of how to adapt a general prompt template for a specific task.**  Seeing how to customize the template for different administrative tasks would help learners who are uncertain whether a general structure applies to their specific need.

# Reviewer Summary

Module 1.4 was the most practically transformative module in the course — the five-part structure with each element mapped to the failure mode it prevents gave me a systematic approach to AI prompting that replaced my previous habit of vague requests, and the exercise produced a noticeably better output that validated the principle on the first test.  The SOP conversion demonstration was the most directly applicable example in the course for my administrative role, and the prompt library artefact is the most useful professional deliverable I have produced in this training.  What should be improved: constraint-writing guidance needs more development — the module correctly identifies vague constraints as a problem but does not give sufficient guidance on writing effective ones; a note advising learners to ensure AI tool access before beginning the exercise would prevent a practical obstacle.  What should stay: the failure-mode mapping for each prompt element (this is the clearest teaching in the module and possibly the course), both demonstrations, the seven-field prompt library template with the Review Notes field, and the accessible language that has made this module as approachable as Module 1.3.
